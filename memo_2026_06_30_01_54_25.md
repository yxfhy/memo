# メモ

deepagents × uv × docker-compose による隔離型「自動開発」環境 — 実践ガイド

TL;DR
- venv/pip から uv への移行は「pip+venv+pip-tools+pyenv+pipx を1バイナリに統合」する乗り換えであり、操作は uv init / uv add / uv sync / uv run / uv lock に集約され、pyproject.toml + uv.lock + .venv が中心になります。Docker では公式イメージ ghcr.io/astral-sh/uv からバイナリをCOPYし、multi-stage build と --mount=type=cache、uv sync --locked、UV_COMPILE_BYTECODE=1 / UV_LINK_MODE=copy / UV_PROJECT_ENVIRONMENT を使うのが定石です。
- deepagents（LangChain純正のagent harness、現行安定版は 0.6.12（2026年6月25日リリース、deepagents[aws] extra 追加。プレリリース 0.7.0a2 もあり）、Python 3.11+）は create_deep_agent 一発で planning（write_todos）/virtual filesystem/subagents/detailed system prompt を備えたエージェントを生成します。エージェントにコード実行をさせるには「sandbox backend」を与え execute ツールを有効化します。公式の sandbox backend は Modal/Daytona/Runloop/E2B/LangSmith/AgentCore のクラウド型のみで、ローカルDockerサンドボックスは自前で SandboxBackendProtocol（BaseSandbox を継承し execute() を実装）として作る必要があります。
- セキュリティ上、エージェントが生成・実行するコードは「敵対的ペイロード」として扱うのが鉄則です（arXiv:2510.12399「A Survey of Vibe Coding with LLMs」は "Because large language models can generate arbitrary and potentially unsafe code, strict security isolation is essential" と明言）。Docker socket のマウント（DooD）や --privileged な DinD は最悪ホストのroot奪取につながるため、本タスクでは「エージェント本体とコード実行コンテナを分離し、実行コンテナは --network none・--read-only・--cap-drop ALL・非root・メモリ/CPU/pids制限で固める」構成（LangChain推奨の "Pattern 2 — Sandbox as Tool"）を推奨します。WSL2 では必ずソースを Linux 側 ext4（~/ 配下）に置き、/mnt/c は避けてください。

Key Findings

uv の本質と移行マッピング
uv は Astral 製の Rust 実装で、「pip, pip-tools, pipx, poetry, pyenv, twine, virtualenv を置き換える単一ツール」を標榜します。速度はAstral公式が "10-100x faster than pip" と表現しており、独立検証ではキャッシュ無し（uncached）で概ね8〜10倍、ウォームキャッシュではハードリンクにより最大100倍程度（例: pip 約38秒 vs uv 約3秒）の差が報告されています。venv/pip ユーザーが押さえるべき対応関係は次の通りです。
- python -m venv .venv → uv venv（uv は仮想環境を前提とし、プロジェクトコマンド実行時に .venv を自動生成・自動利用。 [Astral](https://docs.astral.sh/uv/guides/projects/) activate 不要で uv run 経由で実行可能）
- pip install X → uv add X（pyproject.toml + uv.lock を更新し .venv に反映）。低レベルには uv pip install も使えます。
- pip install -r requirements.txt → uv add -r requirements.txt（移行時）または uv pip install -r
- pip freeze → uv pip freeze、pip-compile（pip-tools）→ uv pip compile、pip-sync → uv pip sync
- pyenv install/local → uv python install / uv python pin（.python-version に書き込み） [Tutorials Technology](https://tutorials.technology/tutorials/uv-python-package-manager-2026.html)
- pipx install / pipx run → uv tool install / uvx（uv tool run）
- 依存のロックと環境同期は uv lock（uv.lock を解決・生成）と uv sync（uv.lock から .venv へインストール、デフォルトで余分なパッケージを削除する exact sync）。uv run は実行のたびに lockfile と環境が最新かを検証します。 [Astral](https://docs.astral.sh/uv/concepts/projects/sync/)

pyproject.toml / uv.lock / .venv の役割
pyproject.toml はプロジェクトのメタデータと依存（[project] dependencies、開発依存は [dependency-groups]）を宣言する標準ファイルで requirements.in を置き換えます。uv.lock は全依存（推移的依存含む）を厳密ピンしたクロスプラットフォームのロックファイルで、人間可読なTOMLですが手編集せず uv が管理します。バージョン管理にコミットして再現性を担保します。.venv はプラットフォーム依存の仮想環境であり、Docker イメージや配布物には含めない（.dockerignore / .gitignore に追加）のが鉄則です。

Python本体のバージョン管理（pyenv不要）
uv はプリビルドのCPython（python-build-standalone）をダウンロードして管理するため、ソースコンパイルする pyenv より高速で、ビルド依存も不要です。 [PyDevTools](https://pydevtools.com/handbook/explanation/how-do-pyenv-and-uv-compare-for-python-interpreter-management/) uv python install 3.12、uv python pin 3.12（.python-version 生成）、uv python list が基本。uv は必要に応じて自動でPythonをDLします。pyenv が入れたPythonは uv からは「system Python」扱いになります。

依存グループ（dev依存など）の管理
PEP 735 の [dependency-groups] を使います。uv add --group test pytest、uv add --dev ruff（= --group dev）。uv sync はデフォルトで本体依存＋dev グループをインストール。--no-dev で dev 除外、--group/--only-group/--all-groups/--no-default-groups で制御。--only-group test はプロジェクト本体もデフォルトグループも除外して当該グループのみ入れるためCIや軽量実行に有用です。tool.uv.default-groups で既定グループを変更でき、conflicts でグループ間の排他（例: 異なる numpy バージョン）も宣言できます。

venv/pip時代の癖で陥りやすい落とし穴
- activate 癖：uv プロジェクトでは基本 activate せず uv run を使う。uv は VIRTUAL_ENV より .venv を優先し、--active で能動環境を優先可。
- pip install を直接叩く癖：uv add を使わないと pyproject.toml と uv.lock が同期しない。低レベル操作が必要なときだけ uv pip を使う。
- requirements.txt 依存の癖：uv add -r で取り込み、以後は pyproject.toml + uv.lock を正とする。export が必要なら uv export --format requirements-txt や uv pip compile を使う。
- uv sync は exact なので「環境に手で入れたパッケージ」は消える点に注意（保持したいときは --inexact）。

uv × Docker / docker-compose 統合
公式ガイドの定石（docs.astral.sh/uv の Docker 統合）：
- uv バイナリは公式イメージからCOPYして導入：COPY --from=ghcr.io/astral-sh/uv:latest /uv /uvx /bin/。 [Astral](https://docs.astral.sh/uv/guides/integration/docker/) 再現性のためバージョンタグ（例 ghcr.io/astral-sh/uv:0.8.x）や SHA256 ダイジェスト固定が推奨。distroless 版（uvバイナリのみ）と derived 版（OS入り、python3.12-bookworm-slim 等）があります。
- multi-stage build + 中間レイヤー分離：依存だけ先に入れる uv sync --locked --no-install-project を1レイヤー、続いてソースをCOPYして uv sync --locked を別レイヤーにすると、依存が変わらない限りキャッシュが効きます。
- キャッシュmount：RUN --mount=type=cache,target=/root/.cache/uv ... uv sync。pyproject.toml / uv.lock は --mount=type=bind で渡すとレイヤーを汚さず高速。
- 主要な環境変数：UV_COMPILE_BYTECODE=1（起動高速化のためバイトコードコンパイル）、UV_LINK_MODE=copy（キャッシュと.venvが別FSのときの警告抑止・確実なコピー）、UV_PYTHON_DOWNLOADS=0（ベースimageのsystem Pythonを使い、uv管理Pythonを誤DLしない）、 [SaaS Pegasus](https://www.saaspegasus.com/guides/uv-deep-dive/) UV_PROJECT_ENVIRONMENT（.venv の場所を固定、system環境へ入れる用途）。
- 本番では --no-editable で .venv だけを最終ステージにCOPYし、ソースを含めない手法も使えます。--locked は uv.lock が最新でなければビルドを失敗させ再現性を保証します（workspace では初回のみ --frozen）。
- docker-compose の開発フロー：compose の develop.watch（action: sync で作業ディレクトリを /app に同期、ignore に .venv/、action: rebuild で pyproject.toml 変更時に再ビルド）が公式推奨。docker compose watch で起動します。Compose Watch は Docker Compose 2.22.0 以降が要件（Docker公式 docs.docker.com/compose/how-tos/file-watch/ "Requires: Docker Compose 2.22.0 and later"）で、Docker Desktop 4.24 でGA（初期alphaは Compose v2.17 / Docker Desktop 4.18 でバンドル）。

.venv をコンテナ内に閉じ込めるパターン
ホストの作業ディレクトリを bind mount しつつ、ホスト側の .venv（プラットフォーム依存）でコンテナ内の .venv を上書きしないことが重要です。公式の手法は docker run --volume .:/app --volume /app/.venv のように、.venv を匿名ボリュームでマスクする方法。docker-compose では named volume を .venv にマウントして同様にマスクします。これにより「ソースはホットリロード、依存環境はコンテナ用を維持」を実現できます。

WSL2 特有の注意点（重要）
- ソースは必ず WSL の Linux ファイルシステム（~/ 配下、ext4）に置く。/mnt/c（Windows側NTFS）に置くと Plan9 経由のクロスFS共有となり、メタデータ操作が極端に遅くなり、ディレクトリ走査が数十秒かかることもあります。
- inotify（ファイル変更イベント）は /mnt/c では伝播しません。watch/ホットリロードを使うなら監視対象を WSL 側に置くことが必須。Linux 側に置いた bind mount なら Docker Desktop は inotify を伝播し、同じカーネル/VFSキャッシュを共有するため高速です。
- 大量の小ファイル（依存ディレクトリ、キャッシュ、DBデータ）は named volume に逃がす（「bind mount はソース用、named volume は大量書き込み用」が鉄則）。これは uv のキャッシュや .venv にも当てはまります。
- 非rootでコンテナを動かし、bind mount への root 書き込みで所有権が壊れるのを避ける。Docker Desktop（WSL2エンジン）でもネイティブ docker engine でも、ボトルネックの大半は「Windows側にファイルを置いていること」に起因します。

deepagents の使い方
deepagents は「batteries-included な agent harness」で、create_agent（LangChain）のより意見の強い上位層です。LangGraph ランタイム上で durable execution / streaming / human-in-the-loop を提供します。4本柱は planning tool（write_todos）、subagents（task ツールで隔離コンテキストの子エージェント）、virtual filesystem（pluggable backend）、detailed system prompt（Claude Code 由来）。 [DataCamp](https://www.datacamp.com/tutorial/deep-agents)
- インストール：uv add deepagents と、使うLLMのプロバイダ（uv add langchain-anthropic / langchain-openai 等）。
- 最小起動：
  from deepagents import create_deep_agent
  def get_weather(city: str) -> str:
      """Get weather for a given city."""
      return f"sunny in {city}"
  agent = create_deep_agent(model="anthropic:claude-sonnet-4-6", tools=[get_weather], system_prompt="You are a helpful assistant")
  agent.invoke({"messages": [{"role": "user", "content": "..."}]})
  既定モデルは Claude Sonnet 系。init_chat_model やプロバイダのChatモデルオブジェクトを model= に渡せます。create_deep_agent はコンパイル済み LangGraph グラフを返すため streaming / checkpointer / Studio が使えます。
- カスタムツール：型ヒントとdocstring付きの普通のPython関数を tools=[...] に渡すだけ。Pydantic が型ヒントからJSONスキーマを生成しLLMに渡します。MCP サーバ経由のツールも [Langchain](https://docs.langchain.com/oss/python/deepagents/overview) langchain-mcp-adapters で接続可能。
- subagents：辞書（name/description/system_prompt/tools/model）で定義し subagents=[...] に渡す。複雑なものは CompiledSubAgent（独自の LangGraph グラフ、state に "messages" キー必須）として渡せます。include_builtin_subagents=False で組み込みの general-purpose を無効化できます。
- system prompt：system_prompt= で渡すと middleware が注入する既定指示に「追記」されます（既定の挙動と矛盾する指示は避ける）。
- LLMバックエンド：tool calling 対応なら何でも可（OpenAI/Anthropic/Google のフロンティアAPI、Baseten/Fireworks のオープンウェイト、Ollama/vLLM のセルフホスト）。 [PyPI](Client Challenge (https://pypi.org/project/deepagents/))
- virtual filesystem と実体の連携：ls/read_file/write_file/edit_file/glob/grep が pluggable backend（StateBackend=メモリ、FilesystemBackend=ローカルディスク、StoreBackend=LangGraph Store、CompositeBackend=パスでルーティング、 [Medium](https://medium.com/@shubham.shardul2019/deep-agents-chapter3-everything-claude-agent-can-do-plus-more-each-feature-explained-with-code-dbebc87f83d7) 各種 Sandbox backend）に対して動きます。permissions= で read/write のパス制限（allow/deny、宣言順first-match）を宣言できますが、sandbox backend には permission は適用されず execute による任意コマンド実行が可能になります。
- 最新状況：現行安定版は deepagents 0.6.12（2026年6月25日、deepagents[aws] extra 追加）。プレリリース 0.7.0a2 あり。v0.4 で pluggable sandbox（Modal/Daytona/Runloop）追加、v0.6 で harness profiles・code interpreter 等。API は活発に変化しているため、固定バージョンで pin して使うのが安全です（不確か：細かいAPIは将来変わり得る）。

Dockerで隔離した「エージェントによるコード実行」のアーキテクチャ
なぜ隔離が必要か：LLMが生成するコードは Remote Code Execution、任意ファイル操作、データ持ち出し、リソース枯渇のリスクを持つ「 [APXML](https://apxml.com/courses/building-advanced-llm-agent-tools/chapter-5-advanced-tool-functionality/code-execution-tools) 信頼できない当事者」です。arXiv:2510.12399「A Survey of Vibe Coding with LLMs」は "Because large language models can generate arbitrary and potentially unsafe code, strict security isolation is essential" と述べ、Agentic AI のセキュリティ文献も Docker/gVisor 等のコンテナ実行で "confining execution vulnerabilities without exposing the host system" を推奨します。deepagents 自身も "Deep Agents follows a 'trust the LLM' model. The agent can do anything its tools allow. Enforce boundaries at the tool/sandbox level, not by expecting the model to self-police"（公式README）と明言しています。

実装アプローチの比較
- パターンA（推奨・Sandbox as Tool）：エージェント本体（deepagents+LLM呼び出し）はホスト/コンテナで動かし、コード実行のみ別の隔離環境へ委譲。APIキーがサンドボックス外に留まり、エージェントコードを再ビルドなしで更新でき、複数サンドボックスで並列実行も可能。LangChain は公式にこの "Pattern 2 — Sandbox as Tool" を推奨しています（現行の本番推奨は third-party sandbox API または LLM プロバイダ API の利用で、旧来の Pyodide+Deno ネイティブ隔離は deprecated）。
- パターンB（Agent in Sandbox）：エージェント自体をサンドボックス内で動かす。ローカル開発を忠実に再現できるが、APIキーをサンドボックス内に置く必要があり、サンドボックス侵害時のリスクとイメージ再ビルドの手間がある。
- DooD（Docker socket マウント、-v /var/run/docker.sock:/var/run/docker.sock）：コンテナがホストのDockerデーモンを操作でき、 [Medium](https://medium.com/@rashmi160403/securely-running-docker-in-docker-a-comprehensive-guide-cb6efb5b745f) 生成コンテナはホスト上の兄弟コンテナになる。だが socket はホストのroot権限への直結であり、侵害されると docker run -v /:/host ... でホストの全ファイルにrootアクセスされ得る。 [DEV Community](https://dev.to/it-wibrc/cicd-security-mistake-are-you-giving-your-build-container-root-access-to-your-server-2jnm) socket マウントはコンテナ世界で最も危険な設定の一つとされます。
- DinD（docker:dind）：デーモンをコンテナ内で動かす。--privileged が必要でホストとの隔離が下がりセキュリティリスクを生む。Docker自身も一般利用は非推奨。
- 自前 docker exec/subprocess：エージェントの execute ツールが docker exec で別コンテナ内実行、あるいは subprocess で実行。subprocess 直叩きは隔離ゼロ（Level 0）で本番不可。

サンドボックスのセキュリティ強化（最小権限）
コード実行コンテナには次を重ねます（defense in depth）：
- ネットワーク遮断：--network none（外部通信を物理的に不能化。学習データ持ち出し・SSRFを防ぐ）
- 読み取り専用FS：--read-only ＋ 書き込みが要る場所だけ --tmpfs /tmp:rw,noexec,nosuid,size=... や作業用 named volume
- ケイパビリティ削減：--cap-drop ALL（必要時のみ最小限 --cap-add）
- 権限昇格禁止：--security-opt no-new-privileges
- 非root実行：USER appuser（Dockerfile）/ user: "1000:1000"（compose）
- リソース制限：--memory / --cpus / --pids-limit（fork爆弾・OOM・DoS対策）、--ulimit
- できれば rootless Docker、さらに高隔離が要るなら gVisor / Firecracker microVM（共有カーネルのCVEで全コンテナが同時侵害され得るため、未検証コードには有効）

既存ツール vs 自前Docker
- 公式 deepagents の sandbox backend：Modal（GPU可・ML向き）、Daytona、Runloop、E2B、LangSmith Sandbox、AgentCore（AWS）。いずれもクラウド型で、execute ツールと ls/read_file/write_file を提供。setup script で環境変数注入やリポジトリclone可。 [LangChain](https://blog.langchain.com/execute-code-with-sandboxes-for-deepagents/)
- 自前ローカルDocker：公式 deepagents にはローカル/自己ホストDockerバックエンドは同梱されていません（GitHub issue #3128「Add Docker sandbox」で要望中、ステータスはオープン）。自分で SandboxBackendProtocol を実装します。実装すべきは BaseSandbox を継承し execute(command, *, timeout=None) -> ExecuteResponse(output, exit_code, truncated) と id プロパティ、必要なら upload_files / download_files（Docker の put_archive / get_archive）だけ。残りの ls/read/write/edit/glob/grep は BaseSandbox が execute() の上に自動実装します。execute ツールはバックエンドが SandboxBackendProtocol を満たすと自動で有効化されます。
- 注意：deepagents には LocalShellBackend と FilesystemBackend もありますが、これらはホスト上で隔離なしに動きます（LocalShellBackend は [Medium](https://medium.com/@shubham.shardul2019/deep-agents-chapter3-everything-claude-agent-can-do-plus-more-each-feature-explained-with-code-dbebc87f83d7) subprocess.run(shell=True) でホストに任意シェル実行＝隔離ゼロ。公式も "grants agents direct filesystem read/write access and unrestricted shell execution on your host. Use with extreme caution" と警告）。本タスクの「安全なサンドボックス化」目的には不適で、必ずDockerで包む必要があります。
- フレームワークごと乗り換える選択肢：pydantic-deepagents（vstorm-co、Pydantic-AIベース）は --sandbox docker フラグでローカルDockerサンドボックスをネイティブ提供します（ [GitHub](https://github.com/vstorm-co/pydantic-deepagents) pip install pydantic-deep[sandbox]）。ただしサードパーティ製です。
- なお sandbox 内でも prompt injection のリスクは残るため、信頼できる setup script の使用、human-in-the-loop（create_deep_agent の interrupt_on）、短命シークレットが推奨されます。 [LangChain](https://blog.langchain.com/execute-code-with-sandboxes-for-deepagents/)

中核タスク（主観評価値×物理測定値のモデリング）への適用

このモデリングで典型的に使う手法・ライブラリ
従属変数＝主観評価（官能スコア、嗜好スコア等）、特徴量＝物理測定値（センサー、機器測定、組成）という構図では、文献上は次が定番です。
- 重回帰（MLR）、PLS回帰（多重共線性が強い分光・センサー特徴に有効。官能のYが複数属性ならPLS2）。実例としてワイン官能データで「官能記述子をX、総合品質をyとするPLS1」「全官能記述子をY、揮発性化合物をXとするPLS2」を用いた多変量回帰研究があります。
- ランダムフォレスト/勾配ブースティング（非線形、相互作用に強い）。発酵刻み唐辛子(FCP)の電子鼻×官能スコア予測研究（PMC12272893）は "The random forest (RF) demonstrated the highest predictive accuracy among support vector machine (SVM), multiple linear regression (MLR), and back propagation neural network (BPNN)" と報告しています（指標は R²・MSE・RMSE・MAE で比較）。
- 特徴量重要度・SHAP（shap ライブラリ、explainer = shap.Explainer(model.predict, X) [Towards Data Science](https://towardsdatascience.com/using-shap-values-to-explain-how-your-machine-learning-model-works-732b3f40e137/) で寄与を定量化）、permutation importance。
- 官能データ特有：ヘドニック尺度（典型的には1〜9、または1〜5の5段階）の順序性、パネリスト間のばらつき・効果。ベータ回帰や混合効果モデル（順序・[0,1]有界応答）が使われる例もあります（例: glmmTMB によるベータ回帰混合モデル）。少数サンプルでは Leave-One-Out CV が定番。 [arxiv](https://arxiv.org/pdf/2502.07400)
- ライブラリ：scikit-learn、statsmodels、（PLSは sklearn.cross_decomposition.PLSRegression）、xgboost / lightgbm、shap、pandas/numpy、matplotlib/seaborn。これらをコンテナの依存（pyproject.toml）に含め、エージェントに使わせます。

依存の分け方（例：pyproject.toml）
[project] では deepagents 本体と LangChain プロバイダ。サンドボックス側の重いML依存（scikit-learn, xgboost, lightgbm, shap, pandas, numpy, statsmodels, matplotlib）は Dockerfile.sandbox 側の依存グループ（例 [dependency-groups] sandbox = [...]）に入れ、uv sync --only-group sandbox でサンドボックスイメージに焼き込みます。これによりエージェント本体イメージを軽量に保てます。

エージェントが反復的にモデル開発を回すワークフロー例
1. ユーザがタスク（例「data/raw のセンサー特徴から官能総合評価を予測するモデルを作り、R²とRMSEをLOOCVで評価、SHAPで重要特徴を出せ」）を投入。
2. エージェントが write_todos で計画（データ読込→前処理→特徴量→モデル候補（PLS/RF/GBM）→評価→改善）。
3. write_file で /workspace にスクリプトを書き、execute（Dockerサンドボックス内）で uv run python train.py を実行。
4. 標準出力（R²/RMSE/CV結果）を読み、results を /workspace に保存。性能が閾値未満なら特徴量・モデル・ハイパラを変えて再実行（ループ）。
5. SHAP図やレポートを生成し、最終結果を要約。重い中間結果はファイルに退避（context offloading）。
必要に応じ subagent（例「modeling-agent」）に学習・評価だけを隔離コンテキストで委譲し、本体は計画と統合に専念させます。

注：sandbox がモデル学習で外部からデータ取得やpip追加DLを要する場合のみ network を限定的に開けます。原則は none のままにし、依存はイメージに焼き込み、データは read-only マウントで渡します。

4) 自前 Docker バックエンド（公式 deepagents 0.6 系、Sandbox as Tool）
公式にはローカルDockerバックエンドが無いため SandboxBackendProtocol を自前実装します。BaseSandbox を継承し execute() を実装すれば、ls/read/write/edit/glob/grep は基底クラスが execute 経由で提供し、 [DeepWiki](Backend Implementations | langchain-ai/deepagentsjs | DeepWiki (https://deepwiki.com/langchain-ai/deepagentsjs/6-backend-implementations)) execute ツールも自動で有効になります。 [langchain](https://docs.langchain.com/oss/python/deepagents/sandboxes)

import docker
from deepagents.backends.protocol import ExecuteResponse
from deepagents.backends.sandbox import BaseSandbox

class DockerBackend(BaseSandbox):
    """既存の隔離コンテナ(docker compose の sandbox)に docker exec で実行を委譲。"""
    def __init__(self, container_name: str = "project-root-sandbox-1"):
        self._client = docker.from_env()
        self._container = self._client.containers.get(container_name)

    @property
    def id(self) -> str:
        return self._container.id

    def execute(self, command: str, *, timeout: int | None = None) -> ExecuteResponse:
        # 非root・作業ディレクトリ固定で実行
        res = self._container.exec_run(
            cmd=["sh", "-lc", command], user="sbx", workdir="/workspace",
        )
        out = res.output.decode("utf-8", errors="replace") if res.output else ""
        return ExecuteResponse(output=out, exit_code=res.exit_code, truncated=False)

from deepagents import create_deep_agent
agent = create_deep_agent(
    model="anthropic:claude-sonnet-4-6",
    backend=DockerBackend(),
    system_prompt=(
        "あなたは官能評価×物理測定値の予測モデルを開発するMLエンジニアです。"
        "/workspace 配下にコードを書き、execute で uv run して学習・評価し、"
        "R²/RMSE を LOOCV で報告し、SHAP で重要特徴を可視化してください。"
        "全ての成果物は /workspace に保存してください。"
    ),
)
result = agent.invoke({"messages": [{"role": "user",
    "content": "data/raw の特徴量から官能総合評価を予測するモデルを作って評価して"}]})

このバックエンドは GitHub issue #3128 のコミュニティ実装と同じ設計です（公式マージ前のため、timeout の尊重、非root、network、リソース制限の作り込みは自分で行う必要があります）。execute はあくまで隔離済みコンテナへの委譲に徹し、ホスト側にはエージェント本体とAPIキーのみを置くのがパターンAの要点です。

Recommendations
1. まず uv へ移行：既存プロジェクトで uv init → uv add -r requirements.txt → uv lock → uv sync。CIや配布では uv sync --locked（または --frozen）で再現性を担保。pyenv は uv python install/pin に置き換え、requirements.txt は廃止して pyproject.toml+uv.lock を正とする。WSL2 ではリポジトリを ~/ 配下（ext4）に clone する。
2. 段階的に隔離アーキテクチャを構築：(a) まずローカルで create_deep_agent + 自前 DockerBackend（network none・read-only・cap-drop・非root・mem/cpu/pids制限の sandbox コンテナへ docker exec 委譲）を最小構成で動かす。(b) 動いたら subagent 化・SHAP/レポート生成・反復ループを足す。(c) GPU学習や本格運用が必要になったら Modal などクラウドサンドボックスへバックエンドを差し替える（パターンAなのでエージェントコードはほぼ不変）。
3. セキュリティの閾値（これを超えたら設計変更）：エージェントが外部ネットワークアクセスを要求し始めたら、none ではなく専用の内部ネットワーク＋egress whitelist に切り替える。未検証/外部モデルやユーザ入力を扱うなら共有カーネルのDockerでは不十分とみなし gVisor / microVM へ。Docker socket マウントや --privileged は本タスクでは採用しない（どうしても必要なら docker-socket-proxy でAPIを最小化し、EXEC=0 等で制限）。
4. 依存設計：エージェント本体イメージ（deepagents+langchain-*）と ML サンドボックスイメージ（scikit-learn/xgboost/lightgbm/shap/statsmodels/pandas）を pyproject.toml の依存グループで分離し、それぞれ --no-group / --only-group で焼き分ける。これでビルドが速く、攻撃面も小さくなる。
5. human-in-the-loop：破壊的操作や外部呼び出しは create_deep_agent の interrupt_on で承認制にする。 [Langchain](https://docs.langchain.com/oss/python/deepagents/overview) シークレットは短命化し、サンドボックスに常駐させない。

Caveats
- バージョンの速い変化：deepagents は 0.6.12 が安定版、0.7.0a2 が出ており、backend/middleware のAPIは活発に変化中です。ここで示したクラス名（BaseSandbox, SandboxBackendProtocol, ExecuteResponse）やメソッド名は 0.5〜0.6 系の情報に基づきます（SandboxBackendProtocol と execute は "Since v0.2" とされ、現行の BaseSandbox 設計は 0.5〜0.6 系。0.5.0 で backend factory パターンが非推奨化、≥0.5.2 で BackendContext→Runtime 変更）。必ず固定バージョンで pin し、移行時は公式ドキュメントとリリースノートで再確認してください（細部は不確か）。
- ローカルDockerバックエンドは公式未提供：issue #3128 はオープンで、提示の DockerBackend はコミュニティ設計の自前実装です。timeout尊重・非root・network/リソース制限はあなたが作り込む必要があります。
- モデル名の表記：検索結果には gpt-5.5 や claude-sonnet-4-6 等、ドキュメント例として現れるモデル識別子が含まれます。実際に利用可能なモデル名・課金は各プロバイダの最新情報で確認してください（一部は将来のリリース例の可能性があり不確か）。
- セキュリティは絶対ではない：Dockerは共有カーネルであり、カーネルCVEや設定ミスで脱獄され得ます。隔離は「深さの選択」であり、扱うコード/モデルの信頼度に応じてレベルを上げてください。最悪ケース（任意コード実行→ホスト奪取→データ持ち出し）を前提に、network none と read-only を初期値とするのが安全側の既定です。
- 官能評価モデルの統計的留意：少数サンプル・パネリスト効果・順序尺度の扱いを誤ると見かけのR²に騙されます。LOOCV/入れ子CV、適切な誤差指標（RMSE/MAE）、必要ならベータ回帰や混合効果モデルの検討を、エージェントの system prompt に明示してください。
